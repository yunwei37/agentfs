\begin{abstract}
	AI agents increasingly rely on speculative execution: exploring multiple solution paths in parallel and committing only a successful one. Because each speculative branch may modify files and spawn processes, this requires isolating both \emph{filesystem state} and \emph{process state} across branches, with atomic commit and rollback semantics. We formalize this as the \emph{branch context} abstraction and present a two-component framework. First, \sys is a FUSE-based filesystem that gives each speculative branch an isolated copy-on-write workspace, with O(1) creation, atomic commit to the parent, and automatic invalidation of sibling branches, all without root privileges. \sys is open sourced in \url{https://github.com/multikernel/branchfs}. Second, \syscall is a proposed Linux syscall that spawns processes into isolated speculative branches with reliable termination, kernel-enforced sibling isolation, and first-commit-wins coordination. Preliminary evaluation of \sys shows sub-350\,$\mu$s branch creation independent of base filesystem size, and modification-proportional commit overhead (under 1\,ms for small changes).
\end{abstract}

\keywords{Agentic exploration, AI agents, Filesystem branching, Copy-on-write, FUSE, Operating system primitives}

\section{Introduction}

Modern AI agents autonomously execute multi-step tasks through a reason-then-act loop~\cite{react}, where each action (shell commands, file edits, package installations) may produce irreversible side effects. When an attempt fails, rolling back these side effects is difficult. Increasingly, agent systems explore multiple solution paths in parallel~\cite{tree-of-thoughts,reflexion,swe-agent}: for example, a software engineering agent may try several candidate fixes simultaneously, committing only the one that passes tests. This \emph{speculative execution} pattern requires isolating two types of state across parallel branches: \emph{filesystem state} (workspace modifications) and \emph{process state} (execution context), with the ability to commit successful branches and discard failed ones.

Existing OS mechanisms do not meet these requirements. Filesystem branching primitives such as OverlayFS~\cite{overlayfs} require root privileges and lack nested branching; Btrfs~\cite{btrfs} is tied to a single filesystem type; device-mapper snapshots~\cite{dmsnapshot} require raw block devices. For process management, cgroups require setup and may require root privileges, while PID namespaces impose unnecessary overhead (Section~\ref{sec:background}). In practice, current agent frameworks resort to ad hoc solutions such as git stashing, temporary directories, or container clones~\cite{langchain,autogpt}, which incur significant overhead and cannot capture all filesystem modifications.

Our key observation is that FUSE-based file-level copy-on-write, combined with Linux mount namespaces, can provide lightweight speculative branching entirely in userspace without root privileges. We formalize the requirements as the \emph{branch context} abstraction (Section~\ref{sec:branch-contexts}) and present a two-component framework:

\begin{enumerate}
	\item \textbf{\sys}: A FUSE-based~\cite{fuse} filesystem that gives each speculative branch an isolated copy-on-write workspace, with O(1) creation, atomic commit to the parent, and automatic invalidation of sibling branches.

	\item \textbf{\syscall}: A proposed Linux syscall that spawns processes into isolated speculative branches with reliable termination, kernel-enforced sibling isolation, and first-commit-wins coordination.
\end{enumerate}

Preliminary evaluation of \sys shows sub-350\,$\mu$s branch creation independent of base filesystem size, and modification-proportional commit overhead (under 1\,ms for small changes).

We make the following contributions:
\begin{itemize}
	\item The design and implementation of \sys, a FUSE-based branching filesystem with copy-on-write isolation, nested branches, and epoch-based sibling invalidation, all without root privileges.
	\item The design of \syscall, a proposed syscall for branch-aware process coordination with reliable termination and first-commit-wins semantics.
	\item A preliminary evaluation demonstrating the low overhead of \sys.
\end{itemize}

\section{Background and Motivation}
\label{sec:background}

\subsection{AI Agent Execution Patterns}

Modern AI agents follow a \emph{reason-then-act} loop~\cite{react}, executing actions that may produce irreversible side effects. Several patterns amplify this challenge by exploring multiple execution paths: Tree/Graph-of-Thoughts reasoning~\cite{tree-of-thoughts,graph-of-thoughts} searches over candidate solutions in parallel, Reflexion~\cite{reflexion} retries tasks after self-critique requiring rollback of prior side effects, and software engineering agents such as SWE-agent~\cite{swe-agent} and OpenHands~\cite{openhands} attempt multiple fixes each producing substantial filesystem changes. These patterns share a common need for \emph{lightweight, isolated workspaces}. Current frameworks use ad hoc solutions: LangChain~\cite{langchain} and AutoGPT~\cite{autogpt} rely on git stashing, temporary directories, or container clones, while Claude Code~\cite{claudecode} provides per-file snapshots that cannot capture changes from shell commands (e.g., \texttt{npm install}) and do not support parallel branching. None of these solutions provide the combination of low overhead, complete coverage of all filesystem modifications, nested branching, and atomic commit that speculative execution demands.

\subsection{Filesystem Branching Requirements}

The core requirements for speculative filesystem branching are: (1)~portability across underlying filesystems, (2)~nested branch hierarchies, (3)~commit-to-parent semantics, and (4)~efficient sibling invalidation. No existing mechanism satisfies all four (Table~\ref{tab:comparison}). OverlayFS~\cite{overlayfs} supports only a single upper layer, lacks commit-to-parent and sibling invalidation, and requires root privileges. Btrfs~\cite{btrfs} supports nested subvolumes but is tied to a single filesystem type and lacks native commit-to-parent operations. Device-mapper snapshots~\cite{dmsnapshot} require raw block devices and have O(depth) read latency for deep snapshot chains. DAXFS~\cite{daxfs} provides branching with commit and sibling invalidation but targets memory-backed storage and is not portable. At coarser granularity, VM snapshot systems~\cite{qcow2,vmware-snapshots,remus,snowflock} have whole-VM overhead with high creation latency; recent serverless snapshot work~\cite{firecracker,faasnap,catalyzer} does not address branching semantics. CRIU~\cite{criu} requires full state serialization and lacks incremental branching. Among related OS abstractions, Speculator~\cite{speculator} pioneered speculative execution with output gating but focused on distributed consistency, while TxOS~\cite{txos} introduced OS-level transactions that are flat (no nesting), short-lived, and require deep kernel modifications; in contrast, our branches persist for arbitrary durations and operate entirely in userspace.

\subsection{Process Management Requirements}

Speculative execution also requires coordinating processes: (1)~reliable termination of all branch processes on commit/abort, (2)~isolation between sibling branches, and (3)~coordinated creation. No existing mechanism satisfies all three without significant complexity (Table~\ref{tab:process-comparison}). Process groups and sessions~\cite{apue} allow group termination, but processes can escape via \texttt{setsid()} or \texttt{setpgid()}. Manual PID tracking via \texttt{/proc} suffers from race conditions. Cgroups can reliably terminate processes but require setup and may need root privileges, and do not prevent signaling between siblings. The \texttt{clone()} syscall~\cite{clone} enables fine-grained resource sharing but requires combining multiple mechanisms. PID namespaces provide complete isolation but impose PID~1 init overhead. These gaps motivate kernel-level support for branch-aware process management that combines reliable termination with lightweight sibling isolation. More fundamentally, Baumann et al.~\cite{fork-hotos} argue that \texttt{fork()} is no longer a good abstraction for modern systems; \syscall is motivated by similar observations, creating lightweight branches with explicit resource sharing via flags (\texttt{BR\_FS}, \texttt{BR\_MEMORY}).

\label{sec:branch-contexts}

A branch context is an isolated execution environment, comprising filesystem state and process state, created for parallel exploration. Branch contexts provide copy-on-write isolation from a shared origin, atomic commit of successful results, and automatic invalidation of sibling contexts. \subsection{Definition}

More precisely, a branch context~$b_i$ encapsulates: (1)~a filesystem view consisting of the origin's files overlaid with a copy-on-write delta layer~$\Delta_i$, and (2)~a process group whose side effects are confined to~$b_i$. Branch contexts are created in sets of~$N$ siblings from a single \emph{frozen origin}, execute independently, and resolve via atomic commit (promoting one context's changes to the parent) or abort (discarding changes).

The term ``branch context'' is deliberately chosen to evoke version control branching, a metaphor familiar to developers, while emphasizing that the abstraction encompasses both filesystem \emph{and} process state, not merely files. Prior work on speculative OS execution introduced related concepts: Tapus et al.~\cite{tapus} proposed ``speculations'' for distributed fault-tolerance, where a single speculative path runs sequentially from a checkpoint and an owner process decides whether to accept results. Branch contexts differ in three key respects: they support \emph{parallel} exploration of~$N$ paths simultaneously, use \emph{competitive} first-commit-wins resolution rather than owner-decided acceptance, and target \emph{local} agent workloads rather than distributed system consistency.

\subsection{Lifecycle: Fork, Explore, Commit}

Every branch context follows a three-phase lifecycle that mirrors the paper title:

\paragraph{Fork.}
$N$ branch contexts are created atomically from a frozen origin. Each context~$b_i$ receives its own copy-on-write filesystem layer and process group.

\paragraph{Explore.}
Each branch context executes independently. Filesystem modifications are captured in the per-context delta layer~$\Delta_i$ via copy-on-write. Processes within a context can spawn sub-processes, modify files, and install packages, all confined to their context.

\paragraph{Commit/Abort.}
When a branch context succeeds (e.g., tests pass), it \emph{commits}: its delta~$\Delta_i$ is applied atomically to the parent. A context may also \emph{abort}, discarding its delta with no effect on the parent or siblings.

\subsection{Core Semantics}

Branch contexts are governed by four key properties:

\begin{enumerate}
	\item \textbf{Frozen origin.} The parent state becomes read-only when branch contexts exist, eliminating merge conflicts by construction. There is no concurrent modification of the origin to reconcile.

	\item \textbf{Parallel isolated execution.} All $N$ contexts run simultaneously, not as sequential retries, and are fully isolated from one another: no context can observe or modify another's state. This enables speculative speedup proportional to available parallelism.

	\item \textbf{First-commit-wins.} Any context may commit; the first to do so wins, and all siblings are invalidated. This single-winner model avoids the complexity of merging results from multiple branches, which would require conflict detection and resolution strategies that are difficult to make general-purpose. It also contrasts with Tapus's owner-decided acceptance~\cite{tapus} and suits AI agent workloads where the orchestrator cannot predict which branch will succeed.

	\item \textbf{Nested branch contexts.} A branch context may itself fork sub-contexts, forming a tree. Each level commits to its immediate parent, enabling hierarchical speculation (e.g., an agent tries three strategies, each of which tries two sub-variants). The parent can then inspect the result, do additional work, and commit upward in turn.
\end{enumerate}

\noindent The following sections describe how we realize branch contexts: \sys (Section~\ref{sec:branchfs}) implements the filesystem dimension, and \syscall (Section~\ref{sec:branch-syscall}) implements the process coordination dimension.

\section{\sys Design and Implementation}
\label{sec:branchfs}

\sys is a FUSE-based filesystem that realizes the filesystem dimension of branch contexts.

\subsection{File-Level Copy-on-Write}

When a file is first modified on a branch, \sys copies the entire file from the base directory upon first modification (or nearest ancestor branch that modified it) into the branch's delta storage. Subsequent reads and writes to that file are served entirely from the delta copy. Unmodified files are resolved by walking up the branch chain to the base directory.

This file-level granularity is a deliberate trade-off: it is coarser than block-level copy-on-write but simpler to implement correctly atop FUSE, and the overhead of copying whole files is acceptable for the small-to-medium files typical of agent workloads.

\subsection{Branch Chain and Resolution}

Branches form a chain from the current branch back through ancestor branches to the base directory. File lookups traverse this chain:

\begin{enumerate}
	\item Check the current branch's delta storage.
	\item If not found, check each ancestor branch in order.
	\item If not found in any branch, serve from the base directory.
	\item If a \emph{tombstone marker} is encountered at any level, the file is reported as deleted.
\end{enumerate}

Tombstone markers are lightweight sentinel files that record deletions, ensuring deleted files do not ``reappear'' from the base directory.

\subsection{Commit and Abort Semantics}

\paragraph{Commit.}
A commit applies a branch's changes atomically to its parent:
(1)~Modified files and tombstones are collected from the committing branch's delta storage.
(2)~Deletions (tombstones) are applied first.
(3)~Modified files are copied from delta storage into the parent's delta storage.
(4)~The parent's epoch counter is incremented, invalidating all sibling branches.
(5)~Memory-mapped regions from invalidated branches trigger \texttt{SIGBUS}.

\paragraph{Abort.}
An abort discards the branch's delta storage without modifying the parent, incurring negligible cost. Sibling branches remain valid, and memory-mapped regions from the aborted branch trigger \texttt{SIGBUS}.

\subsection{Mount-Based Isolation and Interface}

\sys can be used standalone, without the \syscall kernel extension. In standalone mode, branches are created, committed, and aborted via a CLI tool (\texttt{branchfs create/\allowbreak commit/\allowbreak abort}) or control files (\texttt{.branchfs\_ctl}) within the filesystem, requiring no kernel modifications. All mounts of a single \sys daemon share a single branch namespace, with each branch accessible as a virtual directory via \texttt{@branch} paths (e.g., \texttt{/mnt/workspace/@feature-a/}). Per-branch isolation for multi-agent workflows is achieved by placing each agent in its own mount namespace with only its \texttt{@branch} path bind-mounted as the workspace root, preventing agents from accessing each other's branches. When used with \syscall (Section~\ref{sec:branch-syscall}), these operations are instead driven by the kernel via \texttt{FS\_IOC\_BRANCH\_*} ioctls, and mount namespace setup is handled atomically by the syscall.

\section{The \syscall Syscall}
\label{sec:branch-syscall}

While \sys provides filesystem branching, agents must still coordinate processes across branches. We propose \syscall, a Linux syscall that provides this coordination with capabilities difficult or impossible to achieve purely in userspace: reliable process termination (no escape via \texttt{setsid()}), kernel-enforced sibling isolation, memory branching via page tables, and PID preservation on commit.

\subsection{Interface}

The \syscall interface is designed around three principles: \emph{simplicity}, exposing only three operations (create, commit, abort) that mirror the branching lifecycle; \emph{flexibility}, allowing callers to select exactly which resources to branch via composable flags; and \emph{extensibility}, using the \texttt{bpf(2)}-style multiplexed syscall pattern so new operations and flags can be added without changing the syscall signature.

The syscall takes an operation code and a union of per-operation argument structures (Listing~\ref{lst:interface}).

\begin{lstlisting}[caption={The \texttt{branch()} syscall interface.},label={lst:interface},float=tp]
/* Operations */
#define BR_CREATE    1  /* Create branch(es) */
#define BR_COMMIT    2  /* Commit to parent */
#define BR_ABORT     3  /* Discard branch */

/* Flags for BR_CREATE */
#define BR_FS        (1 << 0)  /* Branch filesystem */
#define BR_MEMORY    (1 << 1)  /* Branch memory */
#define BR_ISOLATE   (1 << 2)  /* Sibling isolation */
#define BR_CLOSE_FDS (1 << 3)  /* Close all fds */

/* Generic ioctls for branching filesystems */
#define FS_IOC_BRANCH_CREATE  _IO('b', 0)
#define FS_IOC_BRANCH_COMMIT  _IO('b', 1)
#define FS_IOC_BRANCH_ABORT   _IO('b', 2)

union branch_attr {
  struct {  /* BR_CREATE */
    __u32 flags;
    __s32 mount_fd;     /* workspace fd */
    __u32 n_branches;
    __u64 child_pids;   /* pid_t __user *, output */
  } create;
  struct {  /* BR_COMMIT */
    __u32 flags;  /* reserved */
  } commit;
  struct {  /* BR_ABORT */
    __u32 flags;  /* reserved */
  } abort;
};

long branch(int op, union branch_attr *attr,
            size_t size);
\end{lstlisting}

\subsection{Operations and Flags}

\paragraph{BR\_CREATE.}
Creates \texttt{n\_branches} child processes, each in its own branch. The kernel issues \texttt{FS\_IOC\_BRANCH\_CREATE} on \texttt{mount\_fd} once per child to create filesystem branches, then atomically forks each child into a new mount namespace and bind-mounts the resulting branch fd over \texttt{mount\_fd} using the new mount API (\texttt{open\_tree} + \texttt{move\_mount}). Returns 0 to the parent (with \texttt{child\_pids} filled) and $1..N$ to each child (branch index). All branches created in a single call form an \emph{exclusive group}: only one can commit successfully; others receive \texttt{-ESTALE}. Four flags control branch scope: \texttt{BR\_FS} (required) gives each child a new mount namespace with its branch mounted as the workspace; \texttt{BR\_MEMORY} additionally branches memory via page table copy-on-write, where writes fault into child-owned pages and the parent's pages become read-only; \texttt{BR\_ISOLATE} adds kernel-enforced signal/ptrace barriers between siblings via \texttt{branch\_id} checks; \texttt{BR\_CLOSE\_FDS} closes inherited file descriptors so children re-open files in their branch context.

\paragraph{BR\_COMMIT.}
The kernel first wins the exclusive group race, then issues \texttt{FS\_IOC\_BRANCH\_COMMIT} on the branch fd to apply filesystem changes, commits memory if \texttt{BR\_MEMORY} was set, terminates siblings (which receive \texttt{-ESTALE} on their next commit attempt), and finally performs PID takeover: the committing branch \emph{replaces} the parent similar to \texttt{execve()}, taking over the parent's PID and continuing with its full CPU state.

\paragraph{BR\_ABORT.}
The kernel issues \texttt{FS\_IOC\_BRANCH\_ABORT} on the branch fd to discard filesystem changes, then terminates the branch process. If all branches abort, the parent resumes.

\paragraph{Nested Branches.}
Branches can create sub-branches by calling \syscall again. A commit replaces only the immediate parent and inherits its branch context; it does not automatically cascade further. The parent's parent (if any) must commit separately to propagate changes upward.

\subsection{Usage Example}

Listing~\ref{lst:usage} shows the typical pattern for an AI agent exploring three approaches in parallel. The first branch to succeed commits its filesystem changes to the parent; siblings receive \texttt{-ESTALE} and exit.

\begin{lstlisting}[caption={Parallel speculation with first-commit-wins.},label={lst:usage},numbers=left,xleftmargin=2em,float=tp]
int mnt = open("/mnt/workspace", O_PATH);
pid_t pids[3];
union branch_attr attr = {
  .create = { .flags = BR_FS,
               .mount_fd = mnt,
               .n_branches = 3,
               .child_pids = (uintptr_t)pids }
};
int idx = branch(BR_CREATE, &attr, sizeof(attr));
if (idx == 0) {
  /* Parent: wait for winner */
  while (wait(NULL) > 0);
  /* Winner's changes now visible in base */
} else {
  /* Child: idx is 1, 2, or 3 */
  bool ok = try_approach(idx);
  if (ok) {
    union branch_attr ca = {.commit = {0}};
    int r = branch(BR_COMMIT, &ca, sizeof(ca));
    if (r == -ESTALE) _exit(1); /* lost */
    /* r == 0: FS + process committed atomically */
  } else {
    union branch_attr aa = {.abort = {0}};
    branch(BR_ABORT, &aa, sizeof(aa));
    /* does not return */
  }
}
\end{lstlisting}

\subsection{Parent and Child Semantics}

\paragraph{Parent Read-Only.}
While branches exist, the parent becomes read-only for both filesystem (\sys denies writes) and memory (pages marked read-only, writes return \texttt{-EAGAIN}). This eliminates conflicts: the parent spawns branches and waits; on commit, the winning branch replaces it.

\paragraph{Fork and Threads.}
\texttt{fork()} within a branch is permitted: the child inherits the mount namespace (filesystem changes captured in delta) but has a separate address space (memory not part of branch state). Threads via \texttt{clone(CLONE\_VM)} share both mount namespace and address space, so their changes are part of the branch. On commit or abort, all descendants are terminated; the kernel tracks them for reliable cleanup.

\subsection{Filesystem-Agnostic Integration}

\syscall integrates with branching filesystems entirely through three generic ioctls: \texttt{FS\_IOC\_BRANCH\_CREATE}, \texttt{FS\_IOC\_BRANCH\_COMMIT}, and \texttt{FS\_IOC\_BRANCH\_ABORT}, following the pattern of existing generic ioctls such as \texttt{FICLONE} and \texttt{FIEMAP}. On \texttt{BR\_CREATE}, the kernel issues \texttt{FS\_IOC\_BRANCH\_CREATE} on \texttt{mount\_fd} to obtain branch directory fds, then bind-mounts each into a child's mount namespace using the new mount API (\texttt{open\_tree} + \texttt{move\_mount}). On \texttt{BR\_COMMIT} and \texttt{BR\_ABORT}, the kernel issues the corresponding ioctl on the branch fd as part of the atomic operation sequence. For FUSE-based filesystems like \sys, ioctls are forwarded to the daemon via \texttt{FUSE\_IOCTL}; for kernel-native filesystems like DAXFS~\cite{daxfs}, they are handled directly. Crucially, the \syscall implementation itself contains no filesystem-specific code: adding a new branching filesystem requires only implementing the three ioctls within that filesystem (in userspace for FUSE, in the filesystem module for kernel-native filesystems), with no changes to \syscall or the VFS layer.

\section{Related Work}

\paragraph{Container and Namespace Isolation.}
Linux namespaces~\cite{namespaces} provide various forms of isolation (mount, PID, network, etc.). Containers~\cite{docker} combine namespaces with cgroups~\cite{cgroups} for resource isolation, and systems like gVisor~\cite{gvisor} add additional sandboxing. MVVM~\cite{mvvm} uses WebAssembly-based containers with hardware enclaves for secure, cross-platform AI agent deployment and workspace migration. Our approach is complementary: we use mount namespaces for filesystem isolation with lightweight branch-ID-based signal barriers in \syscall, focusing on lightweight branching and rollback rather than cross-platform portability or secure enclaves.

\paragraph{Intra-Process Isolation.}
Light-weight contexts (lwCs)~\cite{lwc} provide independent protection domains within a process, enabling fast rollback by switching contexts. Dune~\cite{dune} uses hardware virtualization to give user-level processes access to privileged CPU features for sandboxing. Capsicum~\cite{capsicum} introduces capability-based sandboxing for Unix. These systems focus on isolation within or below the process boundary; \syscall instead provides isolation \emph{across} speculative branches with coordinated commit/abort semantics.

\section{Discussion and Future Work}

\paragraph{Network and Device Side-Effects.}
Our current prototype isolates only filesystem state. Speculative branches can still cause irreversible external actions via network calls, IPC, or device I/O. For example, a branch that sends an HTTP request or writes to a database cannot be rolled back by aborting the branch. A complete solution requires \emph{effect gating}: buffering or journaling external side-effects until commit, and discarding them on abort. Possible approaches include interposing on the \texttt{connect()}/\texttt{sendmsg()} syscalls to redirect network traffic through a branch-aware proxy, or using \texttt{seccomp-bpf} to block external I/O entirely within speculative branches. Agent gateway architectures such as Agentry~\cite{agentry}, which mediate all agent-to-external communication through a central gateway, provide a natural interposition point for branch-aware effect gating. Speculator~\cite{speculator} demonstrated output gating for distributed file systems; extending this to general network and device I/O within our framework is an important direction for future work.

\paragraph{Merge Conflicts.}
\sys supports only single-winner semantics via epoch-based invalidation; when one branch commits, all siblings are invalidated. This is sufficient for first-commit-wins speculation but cannot combine results from multiple branches. Future work will explore three levels of merge support: (1)~\emph{file-level union}, where non-overlapping file changes from multiple branches are merged automatically; (2)~\emph{conflict detection}, where overlapping changes to the same file are flagged for the agent or orchestrator to resolve; and (3)~\emph{semantic merge}, where domain-specific merge strategies (e.g., AST-level merge for source code) enable richer combination of branch results.

\paragraph{\syscall Implementation.}
\syscall is currently a design proposal. Implementation requires kernel modifications for branch-aware process termination, optional branch-ID-based signal barriers, and the generic \texttt{FS\_IOC\_BRANCH\_*} ioctl definitions for filesystem integration. The ioctl-based design keeps the syscall itself filesystem-agnostic: branching filesystems implement the three ioctls (create, commit, abort) in their own code, whether in a FUSE daemon or a kernel module. The memory branching (\texttt{BR\_MEMORY}) feature requires additional complexity for COW commit-to-parent semantics. We plan to prototype \syscall on Linux 6.19, initially supporting \texttt{BR\_FS} and \texttt{BR\_ISOLATE}, with \texttt{BR\_MEMORY} as a follow-on effort due to the page table manipulation involved.

\paragraph{Edge Cases and Limitations.}
\sys operates at file-level granularity, which introduces known limitations. Symlinks are copied as-is into delta storage, so a symlink targeting an absolute path outside the branch will resolve to the unbranched file. Hardlinks are broken on copy-on-write: modifying a hardlinked file creates an independent delta copy, losing the link relationship. Special files (FIFOs, Unix domain sockets, device nodes) are not supported in delta storage; agents that create these within a branch will encounter errors. Additionally, if delta storage exhausts available disk space during a branch's lifetime, subsequent writes will fail with \texttt{-ENOSPC}. These limitations are acceptable for typical AI agent workloads, which primarily involve regular files, but should be addressed for general-purpose use.

\paragraph{Integration with AI Agent Frameworks.}
Current AI agents (Claude Code, SWE-agent, OpenHands) primarily checkpoint via files rather than process memory, suggesting that \texttt{BR\_FS}-only mode would cover most current use cases. The \texttt{BR\_MEMORY} flag provides future capability for agents that maintain significant in-memory state (e.g., persistent Python interpreters or large embedding caches). Integration requires minimal changes to existing frameworks: an orchestrator creates branches before dispatching tasks and commits the winner, while agents themselves run unmodified inside their branch's mount namespace.

\section{Conclusion}

Exploratory execution for AI agents requires isolating both filesystem and process state across parallel execution paths, with atomic commit and rollback semantics. We formalize this requirement as the \emph{branch context} abstraction and present two complementary implementations. \sys is a FUSE-based filesystem that provides lightweight copy-on-write branching with O(1) creation and atomic commit-to-parent, all without root privileges. \syscall is a proposed Linux syscall for kernel-enforced process coordination with reliable termination and first-commit-wins semantics, integrating with branching filesystems through generic \texttt{FS\_IOC\_BRANCH\_*} ioctls that keep the syscall filesystem-agnostic. Preliminary evaluation shows O(1) branch creation and modification-proportional commit costs.
